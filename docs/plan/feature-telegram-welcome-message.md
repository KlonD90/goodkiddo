# Feature: Telegram Welcome Message

## Summary

Add a concise Telegram `/start` welcome message for authorized users. The message should help a new user understand how to begin a normal conversation with the assistant, which common workflows are supported, and which control commands are useful before they send their first real request.

## User Cases

- A new Telegram user can run `/start` and immediately understand they can type a normal request instead of learning a special prompt format.
- A returning Telegram user can run `/start` to rediscover useful commands such as `/identity` and `/new_thread`.
- A Telegram user who runs `/start` receives onboarding without needing an agent turn.

## Scope

**In:**

- Telegram `/start` command for authorized users.
- A short onboarding reply that mentions normal text requests, supported files, `/identity`, and `/new_thread`.
- Command-menu registration for `/start`.
- Explicit guarantee that `/start` is control/onboarding text only and does not invoke the agent.

**Out:**

- A multi-step onboarding wizard.
- Per-user onboarding completion tracking.
- Personalized examples generated by the model.
- Changes to access control or user provisioning.

## Design Notes

### Message Shape

The welcome message should be short enough to read in one Telegram screen. It should tell users they can:

- write a normal request in plain language
- send supported files when they want help with documents or data
- use `/identity` to choose a behavior preset
- use `/new_thread` to start a clean conversation

Avoid making the message sound like documentation. It should be practical onboarding text, not a feature list.

### Access and Command Handling

`/start` should run after Telegram caller resolution. With free-tier auto-provisioning, a first-time Telegram chat can be created as a free user and then receive the welcome message.

The command belongs with Telegram/session control behavior, not permission commands. It should be registered in `TELEGRAM_COMMANDS` so Telegram shows it in the command menu.

### Runtime Behavior

`/start` must not invoke the agent, enqueue a turn, alter thread state, or enter stored conversation history. It is a direct bot reply.

If a normal turn is already running or queued, `/start` can still be answered directly because it does not mutate agent state or thread state. This differs from state-changing commands such as identity selection.

## Validation Expectations

Implementation tasks belong in `docs/plans/`, but this feature should eventually be validated against these observable outcomes:

- `/start` returns the welcome message for an authorized user.
- `/start` does not call the agent and does not enqueue a turn.
- `/start` does not appear in stored conversation history.
- Telegram command registration includes `start`.

## Risks and Open Questions

- **Message length:** Too much detail makes onboarding feel like documentation. Keep the first version compact.
- **Command queue behavior:** Since `/start` is read-only, it can be allowed while a turn is running. If that proves noisy in practice, it can adopt the same queue guard as other slash commands later.
- **Localization:** The first version can use the existing default language. Future work may localize the welcome message alongside status text.
